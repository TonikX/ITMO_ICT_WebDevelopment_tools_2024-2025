# Сравнение способов параллельного парсинга сайтов

## Описание программ

---

### `threading_parsing.py`
Программа выполняет парсинг списка сайтов с использованием **потоков** (`threading`).  
Каждый поток скачивает HTML-страницу сайта, извлекает заголовок (`<title>`) и сохраняет данные в базу данных PostgreSQL.

**Особенности метода:**
- Использует стандартные потоки Python (`threading.Thread`);
- Отлично подходит для сетевого парсинга, где основное время уходит на ожидание ответов от сайтов.

---

### `multiprocessing_parsing.py`
Программа выполняет парсинг сайтов с использованием **процессов** (`multiprocessing`). 
Каждый процесс независимо скачивает HTML-страницу сайта, извлекает заголовок и сохраняет данные в базу данных PostgreSQL.

**Особенности метода:**
- Использует процессы (`multiprocessing.Process`), каждый процесс работает в отдельном адресном пространстве;
- Эффективен для задач, нагружающих процессор (CPU-bound задачи);
- Для сетевых задач может давать лишние накладные расходы из-за запуска отдельных процессов, что снижает общую производительность при малом числе запросов.

---

### `async_parsing.py` 
Программа выполняет парсинг сайтов с использованием **асинхронного программирования** (`asyncio`) и библиотеки `aiohttp`.  
Асинхронные задачи скачивают страницы параллельно, обрабатывают HTML и сохраняют результаты в базу данных PostgreSQL в таблицу `async`.

**Особенности метода:**
- Использует `asyncio` для асинхронного выполнения сетевых запросов;
- Минимизирует время ожидания сетевых ответов за счёт кооперативной многозадачности;
- Наиболее эффективен для большого количества сетевых операций с малым временем CPU-обработки.

---

## Таблица сравнения времени выполнения

| Метод                        | Время выполнения (секунд) |
|-------------------------------|---------------------------|

| Процессы (`multiprocessing`)  | 2.370                     |

---

## Выводы

- **Асинхронность (`asyncio`) показала наилучший результат** — всего **1.135 секунд**. Это связано с тем, что для сетевых запросов асинхронное программирование позволяет максимально эффективно использовать время ожидания ответа от сайтов, не блокируя поток исполнения.
- **Потоки (`threading`) оказались немного медленнее** — **1.225 секунд**. Потоки также хорошо подходят для сетевых задач, но из-за переключения контекста между потоками и GIL могут быть чуть менее эффективными, особенно на больших объемах данных.
- **Процессы (`multiprocessing`) оказались самыми медленными** — **2.370 секунд**. Использование процессов привело к большим накладным расходам из-за необходимости создавать отдельные процессы и копировать данные между ними. Для задач, связанных преимущественно с сетевыми запросами, использование процессов неоправданно.
