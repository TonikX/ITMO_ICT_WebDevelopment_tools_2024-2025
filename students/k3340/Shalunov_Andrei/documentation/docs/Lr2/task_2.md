# Задача

Напишите программу на Python для параллельного парсинга нескольких веб-страниц с сохранением данных в базу данных с использованием подходов threading, multiprocessing и async. Каждая программа должна парсить информацию с нескольких веб-сайтов, сохранять их в базу данных.

## Подробности задания

1. Напишите три различных программы на Python, использующие каждый из подходов: threading, multiprocessing и async.
2. Каждая программа должна содержать функцию parse_and_save(url), которая будет загружать HTML-страницу по указанному URL, парсить ее, сохранять заголовок страницы в базу данных и выводить результат на экран.
3. Используйте базу данных из лабораторной работы номер 1 для заполенния ее данными. Если Вы не понимаете, какие таблицы и откуда Вы могли бы заполнить с помощью парсинга, напишите преподавателю в общем чате потока.
4. Для threading используйте модуль threading, для multiprocessing - модуль multiprocessing, а для async - ключевые слова async/await и модуль aiohttp для асинхронных запросов.
5. Создайте список нескольких URL-адресов веб-страниц для парсинга и разделите его на равные части для параллельного парсинга.
6. Запустите параллельный парсинг для каждой программы и сохраните данные в базу данных.
7. Замерьте время выполнения каждой программы и сравните результаты.

## Результаты замеров
| Подход               | Воркеры      | Сохранено книг | Время, сек |
|----------------------|--------------|----------------|------------|
| Threading            | 4 потока     | 100            |  8.63       |
| Multiprocessing      | 4 процесса   | 100            | 7.93       |
| Asyncio + aiohttp    | 4 корутины   | 100            | 7.86       |

## 1. Общая архитектура  

1. **Генерация страниц поиска** — 100 записей, шаг 25 (`tpl = "/ebooks/search/?…&start_index={}"`).  
2. **Сбор ссылок на книги** (функция `parse_book_links`).  
3. **Параллельный обход** полученного списка ссылок:  
    - запрос HTML  
    - `parse_book_details` → словарь с полями  
    - `INSERT … ON CONFLICT DO NOTHING` в Postgres  
4. **Агрегация результатов** и замер времени.

## 2. Файл `parser.py`  

- **SessionPool**  
    - Буферизует `requests.Session()` с авто-ретраями (HTTP 500, 502, 503, 504).  
    - Методы: `get_session()` / `put_session()`.  
- **parse_book_links(html)**  
    - Находит все `<li class="booklink">` → возвращает полный URL.  
- **parse_book_details(html)**  
    - Ищет `<table id="about_book_table">`, проходит по `<tr>`, извлекает заголовки и значения (ссылки vs. plain text).  
    - Защищён от отсутствия таблицы.


## 3. Файл `db.py`  

- **prepare_book_record(raw, source)**  
    - Отбирает поля `title`, `author`, `release_date`, `subject` → парсит дату `%b %d, %Y` → возвращает кортеж `(author, title, date, genre, source)`.  
- **AsyncDBFiller** (`asyncpg`)  
    - `connect()` → создаёт пул  
    - `add_book(...)` → `INSERT … ON CONFLICT DO NOTHING`  
    - `disconnect()` → закрывает пул  
- **SyncDBFiller** (`psycopg2`)  
    - `add_book(...)` — то же, но с блокировкой `lock` для потоков.

## 4. Файл `thread.py`  

- **Параметры**: 4 потока, 100 записей  
- Сбор ссылок: один пул сессий, все страницы поиска обрабатываются последовательно.  
- `worker(links)` в каждом потоке:  
    1. берёт сессию из пула  
    2. делает `GET` → парсит детали → `filler.add_book(...)`  
    3. возвращает сессию обратно в пул и возвращает число успешно сохранённых  

## 5. Файл `multiprocess.py`  

- **Параметры**: до 4 процессов (или `cpu_count()`), 100 записей  
- **init_links()**: в основном процессе собирает все ссылки на книги.  
- Каждый процесс в `parse_and_save(chunk)` создаёт свой `SyncDBFiller` и свою `requests.Session()`.  


## 6. Файл `async.py`  

- **Параметры**: 4 корутины, 100 записей  
- Одна `ClientSession()` на весь цикл.  
- `init_links(session)` асинхронно запрашивает страницы поиска и собирает ссылки.  
- `worker(chunk, session, filler)` по очереди делает `await session.get()` → парсит → `await filler.add_book(...)`.  
- Сбор результатов через `await asyncio.gather(...)`


## 7. Выводы  

- **Asyncio + aiohttp**: показал наилучший результат (7.86 с). Это подтверждает, что для массового I/O-bound парсинга асинхронная модель обеспечивает минимальные накладные расходы на переключение контекста.  
- **Multiprocessing**: занял второе место (7.93 с). Процессы дают реальный параллелизм без GIL, но накладные расходы на создание и переключение процессов делают его чуть медленнее асинхронного подхода.  
- **Threading**: самый простой шаблон, но уступил (8.63 с). GIL и блокирующий I/O ограничивают скорость, хотя для небольших задач и простоты реализации его всё ещё можно использовать.
