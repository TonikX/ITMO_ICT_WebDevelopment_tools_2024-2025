## Задание 1: Сравнение threading, multiprocessing и async для вычислений

### Цель
Сравнить производительность трех подходов к параллельным вычислениям на примере расчета суммы чисел от 1 до 10,000,000.

### Реализации

#### Async версия (`main_async.py`)
```python
import asyncio
import time
from concurrent.futures import ThreadPoolExecutor

def calculate_partial_sum(start, end):
    return sum(range(start, end))

async def main():
    start_time = time.time()
    total = 10_000_000
    num_tasks = 4
    chunk_size = total // num_tasks

    loop = asyncio.get_running_loop()
    with ThreadPoolExecutor() as executor:
        tasks = [
            loop.run_in_executor(executor, calculate_partial_sum, 
                               i * chunk_size, 
                               (i + 1) * chunk_size if i != num_tasks - 1 else total)
            for i in range(num_tasks)
        ]
        results = await asyncio.gather(*tasks)

    total_sum = sum(results)
    print(f"Asyncio result: {total_sum}, Time: {time.time() - start_time:.2f} sec")

if __name__ == "__main__":
    asyncio.run(main())
```


#### Multiprocessing версия (`main_multiprocessing.py`)
```python
import multiprocessing
import time

def calculate_partial_sum(start, end):
    return sum(range(start, end))

def main():
    start_time = time.time()
    num_processes = 4
    total = 10_000_000
    chunk_size = total // num_processes
    
    with multiprocessing.Pool(processes=num_processes) as pool:
        tasks = []
        for i in range(num_processes):
            start = i * chunk_size
            end = total if i == num_processes - 1 else (i + 1) * chunk_size
            tasks.append(pool.apply_async(calculate_partial_sum, (start, end)))
        
        results = [task.get() for task in tasks]
    
    total_sum = sum(results)
    print(f"Multiprocessing result: {total_sum}, Time: {time.time() - start_time:.2f} sec")

if __name__ == "__main__":
    main()
```

**Результат:**
```
Multiprocessing result: 49999995000000, Time: 0.09 sec
```

#### Threading версия (`main_threading.py`)
```python
import threading
import time

def calculate_partial_sum(start, end, result, index):
    result[index] = sum(range(start, end))

def main():
    start_time = time.time()
    num_threads = 4
    total = 10_000_000
    chunk_size = total // num_threads
    threads = []
    results = [0] * num_threads

    for i in range(num_threads):
        start = i * chunk_size
        end = total if i == num_threads - 1 else (i + 1) * chunk_size
        thread = threading.Thread(target=calculate_partial_sum, 
                                args=(start, end, results, i))
        threads.append(thread)
        thread.start()

    for thread in threads:
        thread.join()

    total_sum = sum(results)
    print(f"Threading result: {total_sum}, Time: {time.time() - start_time:.2f} sec")

if __name__ == "__main__":
    main()
```

**Результат:**
```
Threading result: 49999995000000, Time: 0.10 sec
```

### Сравнительный анализ

| Метод          | Время выполнения | Относительная скорость | Лучше всего подходит для |
|----------------|------------------|------------------------|--------------------------|
| Multiprocessing| 0.09 сек         | 1.0x (база)            | CPU-bound задач          |
| Async          | 0.10 сек         | ~1.1x медленнее        | I/O-bound операций       |
| Threading      | 0.10 сек         | ~1.1x медленнее        | Простых I/O задач        |

**Выводы:**
1. Для CPU-bound задач (как вычисление суммы) multiprocessing показывает лучшую производительность
2. Threading и Async показывают схожую производительность благодаря GIL в Python
3. Async подход более эффективен для I/O-bound задач благодаря event-loop

```python
    def get_with_retry(self, url, max_retries=3):
        for attempt in range(max_retries):
            try:
                self.update_headers()
                time.sleep(random.uniform(1, 3))  # Имитация человеческого поведения
                response = self.session.get(url, timeout=15)
                
                if response.status_code == 403:
                    return self._try_alternative_approach(url)
                
                response.raise_for_status()
                return response
            except Exception as e:
                logger.warning(f"Attempt {attempt+1} failed: {str(e)}")
                if attempt == max_retries - 1:
                    raise
                time.sleep(2 ** attempt)  # Экспоненциальная задержка

def thread_worker(urls, result_queue):
```

## Multiprocessing реализация (`main_multiprocessing.py`)

### Архитектура решения
1. **Процессно-ориентированный подход**:
   - Использование отдельных процессов для полного параллелизма
   - Обход ограничений GIL (Global Interpreter Lock)
   - Разделяемая память через Manager для сбора статистики

2. **Механизм распределения задач**:
   - JoinableQueue для безопасной передачи URL между процессами
   - Фиксированный пул worker-процессов
   - Poison pill паттерн для graceful shutdown

3. **Управление соединениями**:
   - Отдельные requests.Session для каждого процесса
   - Автономные DNS-резолверы в каждом процессе
   - Индивидуальные User-Agent для каждого worker'а

### Ключевые компоненты

```python
class ResultStats:
    def __init__(self, manager):
        self.success_count = manager.Value('i', 0)
        self.error_types = manager.dict()  # Потокобезопасный словарь
        self.lock = manager.Lock()  # Синхронизация доступа

def worker(url_queue, stats):
    while True:
        url = url_queue.get()
        if url is None:  # Сигнал завершения
            break
        process_url(url, stats)
        url_queue.task_done()

def main():
    manager = multiprocessing.Manager()
    stats = ResultStats(manager)
    url_queue = multiprocessing.JoinableQueue()
    
    # Запуск worker-процессов
    processes = [
        multiprocessing.Process(target=worker, args=(url_queue, stats))
        for _ in range(min(3, len(urls)))
    ]
    
    # Обработка завершения
    url_queue.join()
    for _ in processes: url_queue.put(None)
    for p in processes: p.join()
```

### Особенности реализации
1. **Безопасность процессов**:

   - Автоматическое разрешение DNS в каждом процессе

2. **Механизмы надежности**:

   - Оптимальное количество одновременных запросов: 50-100
   - Обязательно добавляйте задержки между запросами

2. *Multiprocessing* - для сложного анализа:
   - Когда нужна тяжелая обработка DOM
   - Для ресурсоемкого парсинга PDF/изображений
   - В сочетании с asyncio (процессы для CPU, корутины для I/O)

3. *Threading* - временное решение:
   - Для быстрого прототипирования
   - При работе с legacy-кодом
   - Когда важна простота, а не производительность

### Архитектурные выводы

1. **Масштабируемость**:
   - Asyncio масштабируется лучше (тысячи соединений)
   - Multiprocessing требует больше серверных ресурсов
   - Threading быстро упирается в ограничения GIL

2. **Устойчивость**:
   - Multiprocessing устойчивее к падениям
   - Asyncio требует тщательной обработки ошибок
   - Threading подвержен взаимным блокировкам

3. **Экосистема**:
   - Asyncio: быстро растущий набор библиотек
   - Threading: зрелая экосистема
   - Multiprocessing: ограниченные инструменты IPC

**Заключение**:

Выбор подхода должен определяться природой задачи:
- *Чистый I/O (сетевые запросы)* → Asyncio
- *Тяжелые вычисления* → Multiprocessing 
- *Смешанные задачи* → Комбинация asyncio + процессы
- *Legacy-системы* → Threading

Для современных веб-приложений оптимальной архитектурой становится сочетание asyncio для I/O операций и вынесение CPU-intensive задач в отдельные процессы через очереди (Redis, RabbitMQ). Threading сохраняет свою нишу в специфических сценариях, где важна простота реализации, а не максимальная производительность.
